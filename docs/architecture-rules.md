# Архитектурные правила Idaten

Этот документ обязателен для чтения перед любой итерацией. Он содержит устойчивые
ограничения проекта; детали конкретного релиза находятся в `docs/iterations/`.

## 1. Границы системы

Idaten принимает данные из разных источников, но бизнес-логика работает только с
универсальной сущностью `Activity`. Источник отвечает за получение и нормализацию,
но не за аналитику, рекомендации или публикацию.

Целевой поток:

```text
input -> raw artifact -> source adapter -> normalized activity
      -> validation -> deduplication -> persistence -> analytics
      -> coach rules -> optional wording provider -> publication
```

Не все стадии существуют в MVP 0.1, однако новые реализации не должны делать их
последующее выделение невозможным.

## 2. Слои и направление зависимостей

```text
Telegram / HTTP / jobs
        -> application services
        -> repositories + domain calculations
        -> SQLAlchemy / external adapters
```

- Telegram handlers — транспортный слой. Они разбирают ввод, вызывают сервис и
  форматируют ответ. SQL-запросы, расчет статистики и coach rules там запрещены.
- API routers следуют тому же правилу и не являются «настоящим backend» сами по себе.
- Application services управляют use case и транзакцией.
- Репозитории инкапсулируют запросы к БД.
- Analytics — чистые или детерминированные вычисления. LLM не считает метрики.
- Интеграции зависят от доменного контракта, домен не зависит от интеграций.

## 3. Данные и единицы измерения

- Расстояние хранится целым числом метров.
- Время хранится целым числом секунд.
- Пейс хранится как секунды на километр и вычисляется детерминированно.
- Временные отметки timezone-aware. Пользовательская статистика учитывает IANA timezone.
- Идентификаторы внутренних сущностей — UUID.
- Enum-значения стабильны и записываются в БД строками.
- Raw payload не заменяет нормализованные колонки.
- GPS/HR series не раскладываются по SQL-строкам: файл в storage, summary в БД.

## 4. Privacy by default

- Новая активность по умолчанию `PRIVATE`.
- Маршрут, координаты, пульс, точное время старта и raw payload не публикуются в группу
  без отдельного явного разрешения.
- Импорт Strava API по умолчанию private-only, не участвует в групповых рейтингах и не
  передается внешней LLM.
- Проверка разрешений выполняется на backend перед публикацией, а не только в UI.
- Секреты, токены и персональные payload запрещено писать в логи.

## 5. Надежность и эксплуатация

- Production default — PostgreSQL и local filesystem; Redis/MinIO не обязательны.
- Один deployable backend допустим до появления измеримой причины для разделения.
- Повторная доставка webhook/job не должна создавать дубликат.
- Миграции Alembic — единственный способ менять production schema.
- `/health` проверяет процесс, `/ready` — доступность обязательных зависимостей.
- Конфигурация идет через environment variables; `.env` не коммитится.
- Background-задачи должны иметь ограниченные retry и наблюдаемое состояние.

## 6. Coach и LLM

- Rules first, LLM second.
- LLM получает только подготовленный `facts_json`, не raw activity.
- Без ключа LLM весь основной продукт продолжает работать на шаблонах.
- Медицинские диагнозы и гарантии результата запрещены.
- Рекомендации не должны резко увеличивать недельный объем.

## 7. Качество изменений

Каждая итерация обязана:

1. Не расширять scope без изменения спецификации и decision log.
2. Добавлять миграции для schema changes.
3. Покрывать критические расчеты и use cases тестами.
4. Обновлять документацию запуска и handoff.
5. Не коммитить секреты, generated cache и пользовательские данные.
6. Завершаться небольшими осмысленными коммитами и draft PR.
